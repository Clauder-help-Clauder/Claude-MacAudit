# Expert 3: Don Norman — 产品哲学与设计心理学审查 Round 3

## 审查焦点
**反馈系统（Feedback）与状态沟通（State Communication）** — 每个操作是否有即时反馈？用户是否知道系统正在做什么？

## 发现的问题

### 问题 1: GUI 审计完成无过渡反馈，直接跳转
- **严重程度**: HIGH
- **设计原则违反**: Feedback（反馈）
- **文件**: `AppViewModel.swift:230-234` + `ScanningView.swift`
- **描述**: `performAudit` 完成后直接设置 `selectedScreen = .results`。用户看着雷达扫描动画，下一秒被弹射到 ResultsView。没有"扫描完成"的视觉确认，没有分数揭示的仪式感，没有"跳转中"的过渡。对比：扫描过程中有非常丰富的反馈（进度百分比、当前模块名、雷达动画、终端日志），但最关键的"完成时刻"反而最沉默。
- **用户心理影响**: 错愕——"等一下，完成了？我的分数呢？" 用户需要几秒钟确认自己在新页面。
- **修复建议**: 添加 1.5-2 秒的"审计完成"过渡屏，显示最终分数（带动画从 0 跳到实际值），然后自动或手动进入 ResultsView。

### 问题 2: InlineFixButton 执行失败的反馈缺失
- **严重程度**: CRITICAL
- **设计原则违反**: Feedback（反馈）
- **文件**: `ResultsView.swift:788-819`
- **描述**: InlineFixButton 执行流程：`FIX` → `···` → `✓`。但如果 `vm.executeCommand(cmd)` 失败了呢？代码中 `await vm.executeCommand(cmd)` 的返回值被忽略，无论成功与否都执行 `isDone = true`。用户看到 `✓`，以为修复成功了，实际上命令可能失败了。这在安全工具中尤其危险——虚假的安全感。
- **用户心理影响**: 虚假安心——"已修复！"但实际没有，安全隐患仍在。
- **修复建议**: 捕获命令执行结果，失败时显示 `✗` 并恢复为 `FIX` 状态。同时显示简短的错误信息。

### 问题 3: CLI 中 ServiceManager 的操作确认不足
- **严重程度**: MEDIUM
- **设计原则违反**: Feedback（反馈）
- **文件**: `ServiceManager.swift:181-200`
- **描述**: 批量禁用/启用服务时，只有 `✓ 已禁用` 的成功反馈。缺少：(1) 操作前"确认要禁用 X 项服务吗？"的确认步骤；(2) 操作失败的反馈；(3) 对"重启后生效"的强调不够——只在启用时提到，禁用时没有说明当前运行的服务仍在运行。
- **用户心理影响**: 过度自信——"我禁用了 Siri，它一定已经停了。"实际上可能需要重启。
- **修复建议**: 禁用操作也标注"（当前运行的服务将在重启后停止）"，并在批量操作前添加确认提示。

### 问题 4: GUI 扫描取消无确认对话
- **严重程度**: MEDIUM
- **设计原则违反**: Feedback（反馈）+ Safety
- **文件**: `AppViewModel.swift:179-184`
- **描述**: `cancelAudit()` 立即取消并跳回 Dashboard，已扫描的部分结果被完全丢弃。用户可能误触取消，或在扫描进行到 80% 时意外退出——所有等待白费。没有"确定要取消吗？已扫描 X 个模块"的确认。
- **用户心理影响**: 后悔+挫败——"我不小心按了取消，现在要重来。"
- **修复建议**: 添加确认对话，提供"继续扫描"和"取消并丢弃"两个选项。

### 问题 5: GUI 中修复命令复制缺乏操作确认反馈
- **严重程度**: LOW
- **设计原则违反**: Feedback（反馈）
- **文件**: `DetailView.swift:270-284`
- **描述**: 点击 `[ COPY FIX SHELL COMMAND ]` 后，按钮文字变为 `[ COPIED ]`，2秒后恢复。但复制内容会自动添加 `sudo` 前缀（即使原始命令不需要 sudo，见 line 276）。这个"静默修改"没有在反馈中说明——用户以为复制的是原始命令，实际复制的是加了 sudo 的版本。
- **用户心理影响**: 惊讶——"粘贴后发现多了个 sudo，这是怎么回事？"
- **修复建议**: 如果自动添加了 sudo，在按钮反馈中明确提示，如 `[ COPIED (with sudo) ]`。

### 问题 6: ScanningView 进度反馈的精度问题
- **严重程度**: LOW
- **设计原则违反**: Feedback（反馈）
- **文件**: `ScanningView.swift:101-135`
- **描述**: 进度条标签写 "KERNEL INTEGRITY"，但实际进度是按模块数计算的（`completedModules / totalModules`），与内核完整性无关。4 个分段矩形也暗示 4 个阶段，但实际是 12 个模块。标签与实际进度含义不匹配。
- **用户心理影响**: 误导——"内核完整性扫描完成了 50%？" 实际上是 6/12 个模块完成。
- **修复建议**: 改为 "MODULE PROGRESS" 或 "AUDIT PROGRESS" 等更准确的标签。

## Tacit Knowledge 审查

1. **CLI vs GUI 反馈一致性差距**：CLI 的 ServiceManager 提供了丰富的终端反馈（状态 badge、待操作计数、操作成功消息），但 GUI 的 ServiceCheckRow 执行服务切换后只有 `···` 到按钮文字变化的反馈，没有操作结果的文字确认。两个界面面对的是同一用户，反馈体验不应有如此大的差异。

2. **终端日志窗口的安抚效应**：ScanningView 的 `AUDIT_LOG_STREAM` 终端窗口是一个优秀的设计决策——它提供了"系统正在工作"的连续反馈流，让等待变得可忍受。但日志中 "failed: 0" 用绿色高亮、有失败用灰色，这个颜色映射可能让用户误以为"失败被隐藏了"。

3. **等待中的不确定性管理**：扫描时 "开发工具扫描 — 最长约 60 秒" 是唯一的时间预估提示（`ScanningView.swift:86`）。其他模块没有任何时间预估。不确定性是焦虑的主要来源——用户需要知道"还要等多久？"
