# Expert 1: Steve Krug — UX可用性审查 Round 2

## 审查焦点
交互反馈与状态传达：扫描状态清晰度、进度传达准确性、错误状态指引、操作反馈及时性。

## 发现的问题

### 问题 1: 扫描进度只有模块级粒度，缺乏检查项级反馈
- **严重程度**: HIGH
- **文件**: `AppViewModel.swift:198-226`
- **描述**: `scanProgress` 基于 `completedModules / totalModules` 计算，12个模块中每个模块占8.3%。这意味着即使当前模块包含40+个检查项，进度条也只在模块完成时才跳一次。某些模块（如开发工具）执行时间长达60秒，期间进度条完全静止。
- **用户影响**: 用户在等待约60秒内看到进度条卡在某个百分比不动，会认为程序卡死或崩溃。`currentScanningModule` 虽然显示了当前模块名，但进度条停滞与此形成矛盾反馈——"你说在扫描，但进度条为什么不动？"。
- **修复建议**: 将进度计算细化到检查项级别：`scanProgress = (completedModules + currentModuleProgress) / totalModules`，或在长时模块（如开发工具）内添加子进度指示。

### 问题 2: 扫描过程中没有取消按钮
- **严重程度**: HIGH
- **文件**: `ScanningView.swift` (全文)
- **描述**: ViewModel 有 `cancelAudit()` 方法（`AppViewModel.swift:179-184`），但 ScanningView 的整个界面中没有提供取消按钮。用户启动扫描后，如果发现选错了设置，无法中止操作。唯一的方式是关闭整个窗口。
- **用户影响**: 用户被迫等待完整扫描结束（可能数分钟），即使已经意识到需要更改设置。这违反了"用户应始终有退出路径"的基本可用性原则。
- **修复建议**: 在扫描界面底部或进度条附近添加 "CANCEL" 按钮，绑定 `vm.cancelAudit()`。

### 问题 3: 扫描完成→结果页面的跳转缺乏过渡反馈
- **严重程度**: MEDIUM
- **文件**: `AppViewModel.swift:231-234`
- **描述**: 扫描完成后，代码直接设置 `selectedScreen = .results`，没有任何完成提示。用户可能在看扫描动画时突然被切换到结果页面，感到突兀。没有"扫描完成"的摘要通知（如"发现X个问题"）。
- **用户影响**: 用户可能未注意到页面已经切换，尤其是扫描时间较短时。错过了"扫描刚完成"的关键时刻意味着用户需要在结果页面自行判断数据是否是新的。
- **修复建议**: 添加1-2秒的完成过渡画面（显示总分+关键发现），或在跳转后显示一个轻量通知（toast/banner）："Scan complete — 3 critical issues found"。

### 问题 4: FIX 按钮执行后缺乏明确的成功/失败反馈
- **严重程度**: HIGH
- **文件**: `ResultsView.swift:788-819` (InlineFixButton)
- **描述**: 点击 FIX 按钮后，状态变化为 `···`（执行中）→ `✓`（完成），但 `✓` 只显示2秒就消失了。没有确认修复是否真的成功——`executeCommand` 的返回值被丢弃（`_ = await executor.run(cmd)`），错误被静默吞掉。用户看到的 `✓` 可能对应一个实际失败的命令。
- **用户影响**: 用户以为修复成功了，但实际命令可能因权限不足、文件不存在等原因失败。刷新模块后问题依然存在，造成困惑和不信任。
- **修复建议**: 捕获命令执行结果，成功显示 `✓ FIXED`，失败显示 `✗ FAILED` 并提示原因。不要在2秒后自动隐藏结果。

### 问题 5: Detail 页面的 "COPY FIX SHELL COMMAND" 自动添加 sudo 的行为不可预测
- **严重程度**: MEDIUM
- **文件**: `DetailView.swift:276`
- **描述**: 复制修复命令时，代码自动判断 `fixCmd.hasPrefix("sudo ")`，如果不以 sudo 开头就自动添加 `sudo`。但某些命令不需要 sudo 就能运行（如用户级别的 defaults write），自动添加 sudo 反而改变了命令的语义。
- **用户影响**: 用户粘贴到终端后可能被提示输入密码，增加不必要的摩擦。或者 sudo 执行了一个本应以用户身份运行的命令，导致文件权限错误。
- **修复建议**: 不要自动添加 sudo。如果认为需要 sudo，在 UI 中明确标注（已有 `!SUDO` 标记），让用户自行决定。

### 问题 6: Detail 页面 result/check 为 nil 时只有一个模糊的错误提示
- **严重程度**: MEDIUM
- **文件**: `DetailView.swift:106-108`
- **描述**: 当 `vm.result(for: checkId)` 或 `vm.check(for: checkId)` 返回 nil 时，页面仅显示 "Check not found"，没有任何进一步指引。用户不知道为什么找不到——是数据被清除了？是 checkId 错误？还是扫描还没完成？
- **用户影响**: 用户遇到此状态时完全不知道如何继续，只能通过侧边栏手动导航回去。
- **修复建议**: 提供更详细的错误信息："This check was not found. It may have been removed or the audit data has been refreshed." 并提供 "Return to Results" 按钮。

### 问题 7: 服务模块的 ENABLE/DISABLE 操作无确认对话框
- **严重程度**: HIGH
- **文件**: `ResultsView.swift:873-883` (ServiceCheckRow)
- **描述**: 点击 DISABLE 按钮后直接执行 `launchctl disable` 命令，没有任何确认步骤。虽然这只是 launchctl 操作，但禁用某些系统服务（如 Spotlight、蓝牙相关服务）可能导致系统功能异常。ENABLE 同样是直接执行。
- **用户影响**: 用户可能误触 DISABLE 按钮，立即导致某个系统服务被禁用。没有撤销操作的入口（虽然可以再点 ENABLE，但用户可能不知道哪个被误操作了）。
- **修复建议**: 对关键系统服务添加确认对话框："Disable [ServiceName]? This may affect system functionality." 非关键服务可以保持直接操作。

## Tacit Knowledge 审查

1. **进度条的"最后10%陷阱"**: 当用户看到进度从0%快速增长到80%然后突然减速时（因为最后几个模块检查项更多），心理等待时间会被放大。12个模块的均匀进度计算加剧了这个问题——前几个模块可能很快（<1秒），而开发工具模块要60秒，导致进度在~90%处长时间停顿。

2. **FIX 按钮的"一键修复"陷阱**: 绿色的 FIX 按钮在深色背景上非常醒目，用户可能本能地点击它而不阅读命令内容。由于没有确认步骤，且命令执行后被静默处理，用户形成了"点FIX→问题消失"的简单心智模型，忽略了命令的实际影响。这是一个严重的信任风险——如果某个 FIX 命令有副作用，用户会在不知情的情况下执行它。

3. **扫描日志的颜色编码过于隐晦**: 日志中只有 `failed: 0` 的行显示绿色，其他都是灰色。但用户最关心的是 `failed: N`（N>0）的行——这些行也是灰色，不突出。关键异常信息应该用更醒目的颜色。
