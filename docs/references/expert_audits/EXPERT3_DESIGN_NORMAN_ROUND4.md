# Expert 3: Don Norman — 产品哲学与设计心理学审查 Round 4

## 审查焦点
**情感设计（Emotional Design）与复杂度管理（Complexity Management）** — 工具是否让用户感到安心？复杂信息是否被合理组织？

## 发现的问题

### 问题 1: 纯黑配色 + "THREATS_DETECTED" 语言引发不必要的焦虑
- **严重程度**: HIGH
- **设计原则违反**: Emotional Design — 行为层（Behavioral）+ 反思层（Reflective）
- **文件**: `Tokens.swift:7`（voidBase #0A0A0C）+ `ResultsView.swift:58`
- **描述**: 整个 UI 基于近乎纯黑（#0A0A0C）的背景，配合 `THREATS_DETECTED`、`CRITICAL THREATS`、`VULNERABILITIES`、`CRITICAL RISK` 等大量军事/安全威胁语言。这对安全专业人员可能很酷，但对普通 Mac 用户来说，即使只是一些默认配置差异（如 "未启用防火墙"），也会被渲染为"严重威胁"。情感设计的三层次分析：本能层——暗黑+红色让人紧张；行为层——大量威胁词汇提升焦虑；反思层——"我的电脑很不安全？"的持续不安。
- **用户心理影响**: 焦虑、过度反应——用户可能对非关键问题进行不必要的修改，或在审查前就感到不安。
- **修复建议**: 提供情感层级映射：真正的安全风险（如开放端口、弱密码）使用强烈措辞；配置优化建议（如动画速度、浏览器隐私设置）使用中性语言如"可优化"、"建议调整"。

### 问题 2: Design Tokens 的颜色对比度不足，增加视觉疲劳
- **严重程度**: MEDIUM
- **设计原则违反**: Emotional Design — 本能层（Visceral）
- **文件**: `Tokens.swift:26-28`
- **描述**: `textGhost`（#5A6B52）、`outlineVariant`（#3C4B35）等颜色在 `voidBase`（#0A0A0C）背景上的对比度极低。大量 UI 辅助文字使用这些颜色（如 LAST RESULTS 按钮、模块卡片中的 pass/total 数、时间戳），用户需要眯眼才能辨认。长时间使用会造成视觉疲劳，违背"让用户感到舒适"的情感设计目标。
- **用户心理影响**: 疲惫——"看不清，算了不看了"。
- **修复建议**: 将最低文字对比度提升至 WCAG AA 标准（4.5:1）。`textGhost` 在 #0A0A0C 上的对比度约为 3.2:1，需提升亮度。

### 问题 3: 报告生成器缺乏情感语境——纯数据无解读
- **严重程度**: MEDIUM
- **设计原则违反**: Emotional Design — 反思层（Reflective）
- **文件**: `ReportGenerator.swift:35-44`
- **描述**: Markdown 报告的摘要部分只有冰冷的数字表格（"总计 X，通过 Y，失败 Z"），没有任何解读。用户（尤其是非技术用户）看到"失败: 47"会恐慌，但不知道其中 40 个可能是"动画速度不是最优"这类低风险项。报告缺少一个总评（如"您的系统整体安全状况良好，有 7 个建议优化项"）来提供情感框架。
- **用户心理影响**: 恐慌——"47个失败？！我的电脑是不是被黑了？"
- **修复建议**: 在报告顶部添加一段总评（如 CLI 的 `printOverallSummary` 但带情感语境），区分"安全风险"和"优化建议"。

### 问题 4: CLI 的 InteractiveUI 信息密度过高，缺乏层次
- **严重程度**: MEDIUM
- **设计原则违反**: Complexity Management
- **文件**: `InteractiveUI.swift:35-76`（printResult）+ `123-193`（printFailureSummary）
- **描述**: 每个检测结果行包含：状态符号 + 风险标签 + 检测项名 + 实际值，全部在一行中。在 `printFailureSummary` 中，所有失败项按风险排序，但没有任何分组（如"网络风险"、"隐私风险"、"性能优化"）。用户面对 47 个失败项的平铺列表，缺乏信息层次。
- **用户心理影响**: 信息过载——"太多了，我不知道从哪开始。"
- **修复建议**: 按模块分组显示失败项（如同 GUI 的做法），或至少按风险级别分节。

### 问题 5: HistoryView 空状态的情感传达过于消极
- **严重程度**: LOW
- **设计原则违反**: Emotional Design — 本能层
- **文件**: `PlaceholderViews.swift:28-57`
- **描述**: HistoryView 的空状态使用 `clock.badge.questionmark` 图标和 "NO FIX RECORDS" 文字，配合极低对比度的 #3C4B35 颜色。这传达了一种"你还没有做过任何修复"的消极感，而非"你的系统可能已经很好"的积极感。
- **用户心理影响**: 轻微不安——"我是不是应该修复点什么？"
- **修复建议**: 添加"如果你的审计结果全部通过，这里不会有记录"的说明，或使用更中性的图标（如 `checkmark.shield`）。

## Tacit Knowledge 审查

1. **"Spectral Minimalism"设计系统的情感调性**：设计 token 命名（voidBase、ghostBorder、neonGreen）暗示了一个"赛博朋克安全实验室"的美学愿景。这个愿景对技术用户有吸引力（"酷！"），但对更广泛的 Mac 用户群体可能产生"这不是为我设计的工具"的距离感。设计系统的情感调性应与目标用户的情感需求对齐。

2. **CLI 的"按回车返回"模式**：`waitForReturn()` 是一个经典的 CLI 模式，它给了用户"控制节奏"的权利——这是一种好的情感设计。用户可以在查看结果后自行决定何时继续，而非被系统推着走。GUI 缺少类似的"节奏控制"——扫描完成后立即跳转，没有给用户喘息的空间。

3. **复杂度的渐进式披露**：SettingsView 是渐进式披露的优秀实践——高级设置（版本选择、设备类型、跳过管理）被整齐地组织在独立的页面中，不干扰主要工作流。但 ResultsView 的 CheckListView 缺乏类似的渐进披露——所有 FAIL/WARN/PASS/INFO 项一次性展示，没有折叠/展开机制。
